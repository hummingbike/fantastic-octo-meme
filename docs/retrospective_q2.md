"만들기"보다 "알리기·연결하기"에 시간을 재배분해야 한다.

## 2. 잘한 것

- **산출물 완성도**: W14–W25 전 주차 계획 완료(W23 제외). 단위 테스트 1,355개+
  통과, CI 3.10/3.11/3.12 매트릭스 유지, 릴리즈 파이프라인(태그→테스트→Release
  →PyPI)이 실제로 동작함을 v0.1.0으로 입증.
- **UNKNOWN 원칙 준수**: SynthID 등 탐지 불가 케이스를 과대주장 없이 UNKNOWN
  처리 — 신뢰 자산 훼손 없음. 이번 지표 수집 모듈도 같은 원칙 적용.
- **규제 모니터링 자동화(W24)가 실제 가동**: 매주 월요일 실행되어 상태 커밋
  4회 이상 수행. 자동화 자체는 살아 있다.
- **검증 우선 전략 유지**: 마커(Q3)보다 검증기를 먼저 완성해 법·표준 논의
  (TTA 제안서)에 실측 데이터를 근거로 쓸 수 있는 위치 확보.

| 문제 | 원인 | 교훈 |

| npm 미게시 | JS SDK는 완성됐으나 npm 계정·토큰 수동 절차 미진행 | W16 수동 작업 목록과 함께 일괄 처리할 것 |

### 운영 이슈 상세: 규제 모니터 거짓 양성 (W25에서 수정)

- 증상: 6/22, 6/29, 7/6 매주 "규제/표준 공지 변경 감지" 이슈 자동 생성.
- 실체: 실제 규제 변경이 아니라 `www.tta.or.kr` fetch 타임아웃으로
  `check_regulation_updates.py`가 크래시 → 워크플로의 `|| echo changed=true`
  가 비정상 종료를 변경 감지로 처리.
- 수정: ① `RegulationMonitor.check`가 fetch 실패를 `error` 필드로 기록하고
  변경으로 간주하지 않음 ② 워크플로가 exit 1(실제 변경)만 이슈 생성, 그 외
  크래시는 워크플로 실패로 표면화 ③ 이슈 #30–#32는 거짓 양성으로 종료.
- 상태 파일에 4개 소스 중 2개(msit, nia)만 기준선이 있는 것도 같은 원인
  (tta·law.go.kr은 CI 환경에서 fetch 실패 반복). 이 두 소스는 매 실행
  `FETCH_ERROR`로 기록되어 모니터링 공백이 로그에 드러난다.

## 4. 3분기 방향 제안 (사용자 결정 필요)

PLAN.md Q3(W27–40)는 "마커 + 폐루프" 개발 중심이지만, **채택 0 상태에서 마커를
먼저 만들면 검증기와 같은 문제(만들었지만 아무도 모름)가 반복**된다. 제안:

1. **W16 수동 작업 우선 처리** (블로그·GeekNews·디스콰이엇 게시, npm 게시) —
   Q3 첫 주에 완료. 코드 작업 불필요, 순수 실행만 필요.
2. **주간 시간 배분 변경**: 코드 70% → 코드 50% + 알리기·파트너 컨택 50%.
3. **W23(통합 파트너) 재개**: 갭 리포트(gap_report_v1.md)에 나온 국내 도구
   5곳이 1차 컨택 후보. "귀사 출력물이 제31조를 충족하는지 무료로 검증해
   드립니다"가 훅.
4. 마커 개발(W29)은 위 1–3 진행과 병행하되, 첫 외부 사용자 피드백이 들어온
   뒤 우선순위 재조정.
