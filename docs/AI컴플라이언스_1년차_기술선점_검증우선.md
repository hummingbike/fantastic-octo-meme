## 0. 왜 검증 우선인가 (전략 코어)

- **단독 가치:** 검증기는 *남이 표시한 것*(C2PA 매니페스트·SynthID류·메타데이터·기존 워터마크·가시 라벨 OCR)까지 읽어낸다 → Han의 마커 채택 전에도 즉시 유용.
- **선점 = 기준 정의:** "이 표시가 압축·리사이즈·크롭·재인코딩·캡처를 견디나"를 판정하는 **강건성 벤치마크**를 발표하는 순간, 그 지표의 권위자가 된다.
- **해자 = 네트워크 효과:** 사람들이 Han의 검증기로 확인할수록 → 검증기가 인정하는 방식으로 표시하려는 수요 발생 → Han의 마커·레지스트리로 수렴.
- **표준 정렬:** 한국 표시 표준이 아직 형성 중(TTA 투명성 표준화 진행, 과기정통부 가이드라인 보완 중) → 지금이 **오픈 레퍼런스 구현으로 선점할 창**.
- **유통 = 영업 아님:** GitHub·기술 블로그(Tistory)·TTA 참여·CI 통합으로 개발자가 채택. 콜드 영업 동작 없음.

### 선점이 아닌 것 (시간 낭비 금지)
C2PA 메타데이터 삽입 자체 / 기본 LSB 워터마크 / 가시 라벨 오버레이 → 오픈 표준·라이브러리·상식. 기본기지 해자 아님. **검증의 포괄성·강건성 판정·표준 정렬에만 차별화 투자.**

## 1. 1년차 북극성

- **선점 지표(주):** 오픈 검증 SDK 공개 + 강건성 벤치마크 발표(W16), GitHub 채택(★·설치수), TTA 표준화 컨택/기여(W17~), 호스팅 검증 API 가입수
- **수익(솔직히):** 영업-우선 버전보다 1년차 매출은 **늦고 작다**(호스팅 API+감사 티어 첫 유료 W40 전후, 1년차 MRR 50~200만원 목표). 대신 만드는 자산(검증 해자 + 표준 포지션 + 오픈 채택)이 훨씬 방어적 → 2년차에 마커·레지스트리·영상으로 객단가·MRR 점프.
- **함정 방어:** "검증 없이 빌드"는 엔지니어 함정. 단, 검증기는 standalone 가치라 채택 자체가 곧 검증. **영업이 아닌 "통합 피드백 루프"**(이미지 도구 개발자에게 SDK 써보게 하고 피드백)를 최소로 유지.

## 1분기 (W1–13) — 정찰 · 벤치마크 정의 · 검증기 코어 착수

### Phase 0: 표준·생태계 정찰 + 벤치마크 정의 (W1–4)  ※선점 자산 1호 = 벤치마크
- **W1** — 가이드라인 원문 정독(과기정통부 「투명성 확보 가이드라인」 + TTA 지침서). 제31조 ①고지 ②③표시 요건을 **검증 룰셋**으로 변환(「표시 의무 체크리스트」 A~E를 판정 규칙으로).
- **W2** — 표시 포맷 지형 조사: C2PA 매니페스트 구조, SynthID류 탐지 가능성, EXIF/메타데이터 플래그, 흔한 오픈 워터마크, 가시 라벨. "무엇을 탐지할 수 있고 없는가" 경계 확정.
- **W3** — **강건성 벤치마크 설계.** 변형 배터리(JPEG/WebP 압축률, 리사이즈, 크롭, SNS 재인코딩 시뮬, 스크린샷) × 탐지율 매트릭스. 측정 프로토콜 문서화.
- **W4** — 한국 이미지 도구 샘플 수집·분석(드랩아트·GenApe·벨라·제디터 등 출력물): 실제로 뭘 박나/안 박나, 어떤 변형에서 깨지나 → **갭 리포트 초안**(= 향후 콘텐츠/표준 근거).

> **Phase 0 게이트:** 벤치마크 프로토콜 + 갭 리포트 초안 완성 → 검증기가 "무엇을 판정하는지" 명확하면 진행.

### Phase 1 착수: 검증기 코어 (W5–13)
- **W5** — 인프라 골격(레포/CI), 입력 파이프라인(이미지 디코딩, 포맷 정규화). Han 강점 — 빠르게.
- **W6** — **탐지 엔진 ①:** C2PA 매니페스트 파싱 + EXIF/메타데이터 플래그 판독.
- **W7** — **탐지 엔진 ②:** 가시 라벨 OCR 탐지 + 흔한 오픈 워터마크 탐지 모듈.
- **W8** — **한국법 룰 엔진:** 탐지 결과 → 제31조 충족 여부 판정. 핵심 규칙: 비가시만 있고 "사람 인식 안내"가 없으면 **불충족** 플래그(가이드라인 명시 요건).
- **W9** — **강건성 하니스:** 변형 배터리 적용 → 재탐지 → 생존율 리포트.
- **W10** — 판정 리포트 포맷 확정(콘텐츠별 충족/불충족/생존율/권고). JSON + 사람이 읽는 요약.
- **W11** — CLI 도구 v0 (`verify <image>` → 리포트).
- **W12** — Python/JS SDK 래퍼 + 최소 웹 플레이그라운드(드래그&드롭 검증).
- **W13** — 1분기 회고 + 오픈소스 공개 준비(README, 라이선스, 기여 가이드).

## 3분기 (W27–40) — 마커 추가(검증→마킹 폐루프) · 레지스트리 씨앗 · 첫 수익

- **W27–W29** — **강건 비가시 워터마크 코어.** 모델 무관, 변형 생존을 1순위 목표. 자체 벤치마크로 생존율 측정하며 반복(벤치마크를 이미 보유 = 개발 속도 이점).
- **W30** — **"사람 인식 안내" 자동 동반** 모듈(법 요건) + 가시 라벨 옵션. = 검증기가 "불충족"이라 지적하던 갭을 마커가 해소.
- **W31** — **폐루프 완성:** Verify(갭 탐지) → Mark(우리 마커로 수정) → Re-verify(충족 확인). drop-in API/SDK로 통합.
- **W32–W33** — 마킹 성능·처리량 최적화(비동기 큐, 대량 배치). Han의 파이프라인 강점 영역.
- **W34** — **레지스트리 씨앗:** 발급된 마크를 로그·검증 가능하게 기록(향후 감사·증빙·네트워크 효과 기반).
- **W35** — 콘텐츠 2탄(폐루프·강건성 결과) 공개로 채택·표준 영향력 강화.
- **W36–W37** — 통합 확장: 인기 스택용 플러그인/예제(Next.js, FastAPI, 이미지 생성 파이프라인 훅).
- **W38** — 실사 인물/딥페이크 인접 강화 표시 분기(가상모델 도구 대비) — 단, 분류는 가이드라인 원문 기준(W1 확정분) 준수.
- **W39** — 호스팅 API 안정화 + 첫 유료 전환 제안(대량 검증·마킹·리포트 보관 티어).
- **W40** — 3분기 회고. 채택·API·첫 매출 점검. 가격 재조정.

> **3분기 게이트:** 폐루프 동작 + 호스팅 유료 고객 ≥1 + 레지스트리 가동 → 수익화·확장 모드.

## 리스크 & 균형추

- **채택 없는 빌드 함정:** 검증기 standalone 가치 + 최소 통합 피드백 루프(W17·W24)로 방어. 벤치마크 공개(W16)가 채택 촉발 장치.
- **탐지 한계의 정직성:** SynthID류 등 비공개 마크는 탐지 못 할 수 있음 → 검증기는 "탐지 가능 범위"를 명확히 고지(과대주장 금지). 이 정직성이 오히려 신뢰·표준 권위가 됨.
- **표준 변동:** 룰셋·벤치마크가 표준 보완에 흔들릴 수 있음 → W25·W49 모니터링을 운영 일부로. 변경은 곧 재검증 수요 = 채택 레버로 전환.
- **수익 지연:** 1년차 매출은 작다는 걸 수용. 이 버전의 보상은 매출이 아니라 **방어 가능한 선점 자산**. 현금흐름이 급하면 호스팅 API 유료화(W20·W39)를 앞당겨 조정.
